## 1. 專案概述

**產品名稱**: FixFlow (暫定，後續可更名)

**一句話描述**: 不管是冷氣壞了還是系統掛了，掃碼或填單就能報修，系統自動分級、自動派工、超時自動升級，所有事件一個平台管理。

**技術棧**:
- 前端: React + TypeScript
- 後端: Laravel (PHP)
- 資料庫: MySQL / PostgreSQL
- 通知: LINE Messaging API
- 部署: Railway
- AI (Phase 2): OpenAI API（GPT-4o-mini 便宜、支援圖片辨識，設計抽象層可切換其他模型）

## 2. 要解決的問題

### 2.1 目標使用者

擁有實體設備與/或軟體系統的企業，例如：
- 物業管理公司（管理社區大樓的設備）
- 連鎖門市（每間門市都有 POS、冷氣、冰箱等設備）
- 軟體公司（內部系統需要維護）
- 製造業（工廠設備需要維修排程）

### 2.2 現況痛點

| 痛點 | 說明 | 造成的損失 |
|------|------|------------|
| 不知道壞的是哪台 | 回報者描述不清楚，「3 樓那台冷氣」到底是哪台？ | 維修人員到場才發現找錯設備 |
| 人工分級不準確 | 回報者自己判斷嚴重度，常常該緊急的標成普通 | 重大事件反應慢 |
| 派工靠人喊 | LINE 群組裡喊「誰能處理？」，沒人回就石沉大海 | 工單遺漏、責任不清 |
| 不知道找誰 | 這個問題歸誰管？今天誰值班？都靠記憶或問人 | 轉來轉去浪費時間 |
| 沒有時間壓力 | 沒有超時機制，工單可以放好幾天沒人動 | 客戶/住戶不滿 |
| 設備沒有歷史紀錄 | 這台設備上次修什麼？花多少錢？沒人記得 | 重複維修、無法判斷該不該換新 |
| 無法追溯 | 誰改了什麼、什麼時候回應的，全靠聊天紀錄拼湊 | 管理層看不到真相 |

### 2.3 理想流程（使用本系統後）

**設備報修流程**:
```
設備有問題（冷氣不冷、電梯異常）
  |
  v
[掃 QR Code] 手機打開報修頁面，設備資訊自動帶入
  |
  v
[填寫問題] 只需要描述問題 + 拍照
  |
  v
[自動分級] 系統依規則判定 P0/P1/P2/P3
  |
  v
[自動派工] 查值班表，找到負責的維修人員
  |
  v
[LINE 通知] 維修人員手機收到通知，按「已接手」
  |
  v
[到場維修] 師傅掃 QR Code 確認到場，修完拍照回報
  |
  v
[SLA 追蹤] 限時內未回應 -> 自動升級給主管
  |
  v
[結案歸檔] 維修紀錄存入設備履歷，全程可追溯
```

**系統異常流程**:
```
系統出問題（付款失敗、無法登入）
  |
  v
[Web 表單 / LINE 回報] 描述問題、選擇系統模組
  |
  v
[自動分級] 系統依規則判定 P0/P1/P2/P3
  |
  v
[自動派工] 查值班表，找到負責的工程師
  |
  v
[LINE 通知] 工程師手機收到通知，按「已接手」
  |
  v
[SLA 倒數] 限時內未回應 -> 自動升級給 Team Lead
  |
  v
[解決結案] 全程紀錄可追溯
```

## 3. 產品定位

### 3.1 核心概念

```
進件方式              對象           派給誰         共用引擎
─────────            ────           ──────        ──────────────────────
掃 QR Code       ->  實體設備    ->  維修師傅   ->  自動分級
Web 表單         ->  軟體系統    ->  工程師     ->  自動派工
LINE 文字/照片   ->  都可以      ->  自動判斷   ->  SLA 追蹤
LINE 語音訊息    ->  都可以      ->  自動判斷   ->  超時升級
                                                  稽核紀錄
```

### 3.2 我們 vs 競品

| | 本產品 | 一般 Ticket 系統 | PagerDuty | Notion 串接方案 |
|---|---|---|---|---|
| QR Code 掃碼報修 | 有 | 無 | 無 | 無 |
| LINE 語音報修 | 有（語音轉文字自動建單） | 無 | 無 | 無 |
| 設備履歷追蹤 | 有 | 無 | 無 | 要自己建 |
| 軟體 + 硬體統一管理 | 有 | 只有軟體 | 只有軟體 | 看怎麼設定 |
| LINE 深度整合 | 有（按鈕互動 + 語音） | 無 | 無 | 有限 |
| 自動分級 + 派工 | 有（規則引擎） | 通常手動 | 有 | 無 |
| SLA + 自動升級 | 有 | 部分有 | 有 | 無 |
| 中文 + 台灣在地 | 是 | 看產品 | 英文為主 | 是 |
| 價格 | 親民 | 看產品 | 偏貴 | 便宜但功能有限 |

### 3.3 核心差異化

- **QR Code 是第一公民**: 掃碼報修，零學習成本，設備資訊自動帶入
- **LINE 語音報修**: 傳一段語音訊息就自動建單，打字都不用
- **軟硬體統一管理**: 一個平台處理所有事件，不用分兩套系統
- **全管道進件**: QR Code、Web 表單、LINE 文字、LINE 語音、LINE 照片，全部進同一套引擎
- **設備履歷**: 每台設備的完整維修紀錄，支援決策（修或換）

## 4. 核心功能說明

### 4.1 資產管理 (Asset Management)

每台設備/每個軟體系統都是一個「資產」，統一管理。

**資產類型**:
- equipment (實體設備): 冷氣、電梯、POS 機、印表機、冰箱...
- software (軟體系統): 付款模組、登入系統、後台管理...

**每個資產包含**:
- 唯一編號 (例如 A-012)
- 名稱、型號、位置
- 對應的 QR Code（實體設備才有）
- 負責的團隊
- 上次維護時間
- 完整維修履歷

### 4.2 QR Code 掃碼報修

每台實體設備貼一個 QR Code 貼紙。掃碼後打開手機瀏覽器，進入該設備專屬的報修頁面。

**QR Code 內容**: 一個網址，例如 `https://fixflow.com/report/A-012`

**掃碼後的報修頁面**:

```
+-------------------------------+
|  設備報修                      |
|                               |
|  設備：B棟 3F 大廳冷氣 #A-012  |  <- 自動帶入
|  型號：大金 RXV60UVLT          |  <- 自動帶入
|  位置：B棟 3 樓大廳            |  <- 自動帶入
|  上次維修：2026-02-15          |  <- 自動帶入
|                               |
|  您的姓名 *                    |
|  +---------------------------+|
|  |                           ||
|  +---------------------------+|
|                               |
|  問題描述 *                    |
|  +---------------------------+|
|  | 不冷、有怪聲               ||
|  +---------------------------+|
|                               |
|  拍照（選填）                  |
|  [+] 點擊拍照或上傳            |
|                               |
|  [     送出報修     ]          |
+-------------------------------+

使用者只需要：打幾個字 + 拍一張照
其他全部自動處理
```

### 4.3 Web 表單回報（系統異常用）

非實體設備的問題，透過 Web 表單回報。

**表單欄位**:
- 標題 (必填)
- 問題描述 (必填)
- 系統模組 (必填，下拉選單: Billing / Auth / Order / Infra ...)
- 回報者姓名 / 聯絡方式
- 附件 (選填)

### 4.4 LINE 智慧報修 -- 文字/語音/照片 (Phase 2)

使用者透過 LINE Bot 報修，支援三種方式：

**方式 A：傳文字**
```
使用者：「B棟 3 樓冷氣不冷」
  |
  v
GPT-4o-mini 理解內容，提取：
  設備類型：冷氣
  位置：B棟 3 樓
  問題：不冷
  |
  v
Bot 回問：「請問是大廳的冷氣還是走廊的？」
  |
  v
使用者：「大廳那台」
  |
  v
比對資產資料 -> 確認 #A-012 -> 自動建單
```

**方式 B：傳語音訊息**
```
使用者：（按住錄音）「B棟 3 樓冷氣不冷，好像有怪聲」
  |
  v
LINE Bot 收到語音檔 (m4a)
  |
  v
送到 OpenAI Whisper API -> 語音轉文字
  「B棟 3 樓冷氣不冷，好像有怪聲」
  |
  v
送到 GPT-4o-mini -> 理解內容 + 提取資訊：
  設備類型：冷氣
  位置：B棟 3 樓
  問題：不冷、有怪聲
  |
  v
比對資產資料 -> 確認 #A-012 -> 自動建單
  |
  v
Bot 回覆：
  「已建立報修單 INC-0330-005
   設備：B棟3F大廳冷氣 #A-012
   已通知王師傅，預計今日處理。」
```

**方式 C：傳照片**
```
使用者：（拍設備照片傳到 LINE）
  |
  v
GPT-4o-mini Vision 辨識照片：
  看到冷氣設備、可能看到設備編號貼紙
  |
  v
Bot 回問：「看起來是冷氣設備，請問有什麼問題？」
  |
  v
使用者：「不冷」
  |
  v
自動建單
```

**語音轉文字費用**：OpenAI Whisper API 每分鐘約 US$0.006，一段報修語音 10-30 秒，幾乎不花錢。

### 4.5 自動分級 (Triage)

系統依據規則自動判定事件的嚴重度。

**嚴重度定義**:

| 等級 | 名稱 | 定義 | 設備範例 | 系統範例 |
|------|------|------|---------|---------|
| P0 | 緊急 | 影響安全或核心營運完全停擺 | 電梯困人、消防設備故障 | 全站掛掉、資料庫崩潰 |
| P1 | 嚴重 | 核心功能異常，影響大量使用者 | 整棟停電、大廳空調全壞 | 無法付款、無法登入 |
| P2 | 中等 | 非核心功能異常，影響部分使用者 | 某樓層燈管故障、廁所漏水 | 報表跑不出來、匯出失敗 |
| P3 | 輕微 | 不影響主要功能，可排程處理 | 油漆剝落、門把鬆動 | 文字顯示錯誤、UI 跑版 |

**分級規則引擎（可設定）**:

規則是可設定的，管理員可以自行調整。預設規則範例:

```
設備類相關規則:
  規則 1: 資產類別為「電梯」且描述包含「困人」「卡住」 -> P0
  規則 2: 資產類別為「消防」「電力」 -> 至少 P1
  規則 3: 資產類別為「空調」且描述包含「全部」「整棟」 -> P1

系統類相關規則:
  規則 4: 描述包含「無法付款」「全站掛」「無法登入」 -> P1
  規則 5: 模組為 Billing 或 Auth -> 權重提高（上調一級）

通用規則:
  規則 6: 描述包含「緊急」「立刻」「現在」 -> 權重提高
  規則 7: 以上都不符合 -> P3（預設）
```

### 4.6 自動派工與通知 (Routing & Notify)

**派工邏輯**:

```
事件建立
  |
  v
判斷資產類型:
  有綁定資產 -> 查該資產對應的團隊
  沒有綁定   -> 查事件的模組/分類對應的團隊
  |
  v
查該團隊的值班表 (on-call schedule)
  |
  v
指派給值班人員 + 發送 LINE 通知
```

**LINE 通知內容 -- 設備報修**:

```
[P2] 設備報修 - B棟3F大廳冷氣 #A-012

問題：不冷、有怪聲
位置：B棟 3 樓大廳
回報者：陳小姐
回報時間：2026-03-30 14:00

[已接手]  [需要更多資訊]  [升級處理]
```

**LINE 通知內容 -- 系統異常**:

```
[P1] 系統異常 - 信用卡扣款持續失敗

模組：Billing
描述：多位使用者反映信用卡付款失敗...
回報時間：2026-03-30 14:00

[已接手]  [需要更多資訊]  [升級處理]
```

人員點擊按鈕後，系統透過 LINE Webhook 接收回應，更新事件狀態。

### 4.7 到場確認（設備報修專屬）

維修師傅抵達現場後，掃設備的 QR Code 確認到場。

```
師傅到場
  |
  v
掃設備 QR Code -> 系統記錄到場時間
  |
  v
維修中...
  |
  v
修完 -> 在手機上：描述處理方式 + 拍照
  |
  v
結案 -> 維修紀錄自動寫入設備履歷
```

### 4.8 SLA 倒數與自動升級 (Escalation)

**SLA (Service Level Agreement)**: 服務等級協議，定義每個嚴重度的回應與解決時限。白話說就是「這張工單最慢要在多久內處理完」。

**預設 SLA 規則**:

| 嚴重度 | 首次回應時限 | 解決時限 |
|--------|------------|---------|
| P0 | 5 分鐘 | 1 小時 |
| P1 | 15 分鐘 | 4 小時 |
| P2 | 1 小時 | 24 小時 |
| P3 | 4 小時 | 72 小時 |

**首次回應**: 負責人按下「已接手」的時間。不是解決問題，只是確認「我知道了、我在處理」。

**自動升級流程**:

```
P1 事件建立 (14:00)
  |
  v
等待回應中... SLA 倒數: 15 分鐘
  |
  v
14:10 (剩 5 分鐘) -> Dashboard 顯示黃色警告
  |
  v
14:15 (超時!) -> 自動升級:
  - LINE 通知 Team Lead / 主管
  - Dashboard 顯示紅色
  - 記錄升級事件到 Audit Log
  |
  v
14:30 (超時 15 分鐘, 主管也未回應) -> 再升級:
  - LINE 通知更高層級主管
```

### 4.9 設備履歷 (Asset History)

每台設備的完整維修紀錄，幫助管理者做決策。

```
設備：B棟 3F 大廳冷氣 #A-012
型號：大金 RXV60UVLT
安裝日期：2023-06-15
保固到期：2026-06-15

維修紀錄：
  2026-03-30  不冷、有怪聲    -> 更換壓縮機    王師傅  費用: NT$8,500
  2026-01-12  漏水            -> 清洗排水管    李師傅  費用: NT$1,200
  2025-09-05  不冷            -> 補充冷媒      王師傅  費用: NT$3,000
  2025-06-01  定期保養         -> 清洗濾網      李師傅  費用: NT$800

統計：
  本年度維修次數：3 次
  本年度維修費用：NT$12,700
  -> 建議：維修頻率偏高，建議評估是否換新
```

### 4.10 稽核紀錄 (Audit Log)

每一個操作都會被記錄，不可竄改。

**記錄的事件類型**:
- 事件建立
- 嚴重度變更（含變更前後的值）
- 狀態變更 (new -> triaged -> assigned -> in_progress -> resolved -> closed)
- 指派變更
- 升級觸發
- 通知發送
- 到場確認
- 維修回報
- 備註/留言新增

**範例紀錄**:

```
2026-03-30 14:00:00  [系統]    建立事件 INC-20260330-005
2026-03-30 14:00:01  [系統]    關聯資產: B棟3F大廳冷氣 #A-012
2026-03-30 14:00:01  [系統]    自動分級: P2 (規則: 空調類設備)
2026-03-30 14:00:02  [系統]    自動指派給 王師傅 (空調維修值班)
2026-03-30 14:00:03  [系統]    LINE 通知已發送給 王師傅
2026-03-30 14:05:00  [王師傅]  按下「已接手」(回應時間: 5 分鐘)
2026-03-30 14:30:00  [王師傅]  掃碼確認到場
2026-03-30 15:20:00  [王師傅]  回報完修: 更換壓縮機，費用 NT$8,500
2026-03-30 15:20:00  [系統]    維修紀錄已寫入設備履歷
2026-03-30 15:20:00  [系統]    解決時間: 1 小時 20 分鐘 (SLA: 24 小時, 達標)
```

### 5.2 技術元件說明

| 元件 | 技術 | 說明 |

## 10. AI/Agent 策略

### 10.1 核心原則

**AI 不做決策，AI 做建議。最終判斷由規則引擎或人工確認。**

歷史資料來自人工確認的結果，不是 AI 自己的判斷，確保資料品質。

### 10.2 技術方案

**預設使用 OpenAI API (GPT-4o-mini)**，原因：
- 價格便宜（百萬 token 約 US$0.15 輸入 / US$0.60 輸出）
- 支援圖片辨識（Vision），可以讀報修照片
- 中文能力強
- 生態系成熟，套件多、文件齊全
- 語音功能（Whisper STT + TTS）也在同一個平台，Phase 2 LINE 語音報修直接用

**設計抽象層**，未來可切換：
```
AiService (介面)
  |
  +-- OpenAiService   (預設)
  +-- ClaudeService   (備選)
  +-- GeminiService   (備選)
  +-- OllamaService   (本地開發用)
```

### 10.3 AI 負責的四件事

**1. 結構化摘要（支援照片辨識）**

輸入: 使用者描述 + 報修照片
```
描述：「不冷，有怪聲」
照片：[冷氣出風口的照片]
```

AI 輸出:
```json
{
  "summary": "冷氣不製冷且運轉時有異常噪音，從照片看出風口無明顯結霜",
  "possible_cause": ["壓縮機異常", "冷媒不足", "風扇故障"],
  "missing_info": ["異常聲音是持續還是間歇？", "何時開始出現此狀況？"]
}
```

**2. 建議嚴重度**

```json
{
  "suggested_severity": "P2",
  "confidence": 0.80,
  "reasons": [
    "單一設備問題，非整棟",
    "影響舒適度但不影響安全",
    "非營業核心設備"
  ]
}
```

系統流程: 規則引擎先算出嚴重度，AI 再給建議。兩者不同時取較嚴重的。

**3. RAG (參考歷史工單)**

```
新工單: 「冷氣不冷有怪聲」
  |
  v
搜尋歷史工單: 找到 10 筆「冷氣不冷」的紀錄
  |
  v
AI 參考歷史後判斷:
  「過去 10 筆類似案例中，7 筆最終是壓縮機問題，
   建議優先檢查壓縮機。平均維修費用 NT$6,000-9,000」
```

歷史資料來自人工確認的維修結果，隨使用量增加會越來越準確（Human-in-the-Loop）。

**4. LINE 語音轉文字建單**

```
LINE 語音訊息 (m4a 音檔)
  |
  v
OpenAI Whisper API -> 語音轉文字
  |
  v
GPT-4o-mini -> 理解內容、提取資訊（設備、位置、問題）
  |
  v
比對系統內的資產資料 -> 找到對應設備
  |
  v
自動建單 -> 進入派工引擎

費用：Whisper 每分鐘 US$0.006，一段報修語音 10-30 秒，一天 50 通約 US$0.05
```

全部 AI 功能都在 OpenAI 平台上：GPT-4o-mini（文字理解 + 圖片辨識）、Whisper（語音轉文字），一個 API Key 搞定。

## 11. 商業模式

### 11.1 目標客群

- 物業管理公司（社區大樓設備維護）
- 連鎖門市（各門市設備管理）
- 中小型軟體公司（內部系統維護）
- 設備維修公司（外勤派工管理）

### 11.2 收費方式

| 方案 | 內容 | 月費 (參考) |
|------|------|------------|
| Free | 20 個資產、5 位人員、100 則事件/月 | 免費 |
| Pro | 500 個資產、50 位人員、無限事件、SLA + 統計 | NT$ 3,000-8,000/月 |
| Enterprise | 無限資產、無限人員、AI 加值、SSO、API | 洽談 |

### 11.3 為什麼客戶會付費

能量化的價值:
- **回應時間下降**: 從平均 30 分鐘降到 5 分鐘
- **漏單率下降**: 從每月 10+ 件降到接近 0
- **設備維修成本可視化**: 哪台設備最常壞、該修還是該換，有數據支撐
- **管理透明度提升**: SLA 達標率、各類事件分布，管理層一目瞭然

## 12. 術語對照表

| 術語 | 英文 | 白話解釋 |
|------|------|---------|
| Incident | 事件 | 系統出問題或設備壞了，需要處理的一件事 |
| Asset | 資產 | 被管理的對象，可以是實體設備（冷氣）或軟體系統（付款模組） |
| Triage | 分級/檢傷分類 | 判斷這個問題有多嚴重（醫學術語借用） |
| Severity | 嚴重度 | 問題的嚴重等級（P0 最嚴重，P3 最輕微） |
| SLA | Service Level Agreement | 服務等級協議，約定多久內要回應/解決問題 |
| MTTA | Mean Time To Acknowledge | 平均確認時間，從事件建立到按「已接手」的時間 |
| MTTR | Mean Time To Resolve | 平均解決時間，從事件建立到問題解決的時間 |
| On-call | 值班 | 輪流排班，確保隨時有人可以處理問題 |
| Escalation | 升級 | 問題超時或處理不了，自動/手動上報給更高層級的人 |
| Acknowledge (Ack) | 確認接手 | 表示「我知道了，我來處理」 |
| Routing | 派工/路由 | 決定這個問題要派給誰處理 |
| Audit Log | 稽核紀錄 | 記錄所有操作，事後可追溯、不可竄改 |
| QR Code | 快速回應碼 | 設備上的二維碼，掃描後直接開啟報修頁面 |
| Intake | 進件 | 問題報進系統的過程 |
| Webhook | 回呼網址 | 外部服務（如 LINE）主動把資訊打回你的伺服器 |
| Push Message | 推播訊息 | 伺服器主動發訊息到使用者的 LINE |
| RAG | Retrieval-Augmented Generation | 檢索增強生成，給 AI 參考資料再回答（開書考試） |
| Fine-tune | 微調 | 拿別人的 AI 模型，餵自己的資料再訓練，讓它變更聰明 |
| Human-in-the-Loop | 人在迴圈中 | AI 做建議、人做最終決定，確保品質 |
| RESTful API | -- | 一種設計網頁 API 的規範，用 HTTP 方法操作資源 |
| Migration | 資料庫遷移 | 用程式碼定義資料表結構，可版本控制、可重複執行 |
| Seed | 種子資料 | 預先塞入的假資料，方便開發和 Demo |
| Scheduler | 排程器 | 系統定時自動執行的工作（例如每分鐘檢查 SLA） |
| FK (Foreign Key) | 外鍵 | 資料表之間的關聯欄位，確保資料一致性 |
| PK (Primary Key) | 主鍵 | 每筆資料的唯一識別碼 |
| Enum | 列舉 | 欄位只能填入預先定義的值（例如 P0/P1/P2/P3） |
| SSO | Single Sign-On | 單一登入，用公司帳號就能登入（不用另外註冊） |
| Multi-tenant | 多租戶 | 一套系統同時服務多家公司，資料互相隔離 |
| Vision | 視覺辨識 | AI 看圖片並理解內容的能力 |
| STT | Speech To Text | 語音轉文字，把說的話變成文字（例如 OpenAI Whisper） |
| TTS | Text To Speech | 文字轉語音，把文字變成說的話 |
| Whisper | -- | OpenAI 的語音轉文字服務，支援多語言，費用極低 |

## 附錄: Demo 劇本（面試/展示用）

### 場景 1: QR Code 設備報修（主打功能）

```
步驟 1: 展示一張 QR Code 貼紙（代表貼在冷氣上的）
        用手機掃碼 -> 打開報修頁面
        畫面自動顯示: B棟3F大廳冷氣 #A-012

步驟 2: 填寫問題「不冷、有怪聲」+ 拍一張照 -> 送出

步驟 3: 切換到後台 Dashboard -> 出現新事件
        系統自動分級: P2
        系統自動指派: 王師傅（空調維修值班）

步驟 4: 展示 LINE 通知（手機上）:
        「[P2] 設備報修 - B棟3F大廳冷氣 #A-012」
        按鈕: [已接手] [需要更多資訊] [升級處理]

步驟 5: 按下「已接手」-> 後台狀態即時更新
        MTTA: 3 分鐘

步驟 6: 展示設備履歷: 這台冷氣過去的維修紀錄、費用統計

步驟 7: 展示 Dashboard 統計: SLA 達標率、事件分布
```

### 場景 2: 系統異常 + SLA 升級

```
步驟 1: 在 Web 表單回報「多人無法付款」，模組選 Billing

步驟 2: 系統自動分級 P1 + 指派值班工程師

步驟 3: 展示 SLA 倒數: 15 分鐘內要回應

步驟 4: 模擬 15 分鐘過去（或手動觸發）
        -> 系統自動升級
        -> Team Lead 收到 LINE 通知:
           「[升級] P1 事件未回應 - 無法付款」
```

### 場景 3: LINE 語音報修（Phase 2 Demo，最大亮點）

```
步驟 1: 打開 LINE，對報修 Bot 傳一段語音：

步驟 2: Bot 幾秒後回覆：
        「收到您的報修！
         設備：B棟3F大廳冷氣 #A-012

步驟 3: 切換到後台 Dashboard -> 事件已建立
        展示：語音原文（Whisper 轉的）、AI 摘要、自動分級 P2

步驟 4: 王師傅的 LINE 收到通知 -> 按「已接手」

面試官看到：使用者傳一段語音，幾秒後工單建好、人派好、通知發好
```

### 場景 4: AI 輔助分析（Phase 2 Demo）

步驟 3: 使用者回：「不冷」
